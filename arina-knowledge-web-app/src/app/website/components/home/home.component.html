Semantic Models for business logic, software architecture and information infrastructure: Arina Knowledge

Knowledge Fragmentation

Modern software development suffers from a fundamental problem: knowledge fragmentation. Business requirements live in documents. Process descriptions live in diagrams. Architecture decisions live in presentations. Data models live in databases. Infrastructure lives in deployment scripts. Code lives in repositories.

Each representation describes the same reality from a different perspective, yet none of them is the system itself. As projects evolve, these representations inevitably diverge. Specifications become outdated. Documentation falls behind implementation. Architecture diagrams no longer reflect deployed systems. Business logic becomes buried inside source code.

Traditional software development treats knowledge as text:

  • Requirements Documents
  • Tehcnicals Specifications
  • User stories
  • Tickets
  • Comments
  • Conversations

Knowledge as Code

Knowledge is not text. Knowledge is code. Not metaphorically — structurally:

  • Information becomes knowledge only when it drives decisions and actions.
  • A piece of knowledge should behave consistently under similar conditions.
  • Isolated insights don’t scale. Connected ones do.
  • Knowledge doesn’t run in a vacuum — it runs inside a system with memory.
  • In real environments, you don’t get certainty — you get likelihoods.
  • If knowledge is executable, it must produce outcomes you can evaluate.

Using a GitHub repository as a knowledge base may sound unconventional at first, but it turns out to be surprisingly natural. When knowledge lives in Markdown files inside a Git repository, it automatically gains the same lifecycle as code:

  • Version control: changes are tracked.
  • Collaboration: authors are visible.
  • Traceability: history is preserved.
  • Review processes: discussions happen in pull requests.
  • Branching: improvements are incremental.

Markdown is the simplest text format that is natively supported by GitHub and has several advantages:

  • Human-readable
  • LLM-readable
  • Easy to edit
  • Diff-friendly
  • Platform-independent

Instead of static documentation, knowledge in GitHub repository becomes a living system. Knowledge does not need to be a complicated system. Sometimes the best solution is the simplest one: store knowledge as text, version it, review it, and improve it collaboratively. GitHub repositories already provide the infrastructure for all of this.
